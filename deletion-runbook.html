User Data Deletion Runbook

Irreversible ~5.65 lakh users 2 Databases Shiksha Platform — Inactive User Cleanup

1 — Overview

This operation permanently deletes approximately 5,65,000 inactive users who have a NULL lastLogin value and no activity records in Course, Content, or QuestionSet trackers. Deletion spans two databases and seven tables in total.

🔴 This operation is irreversible. An S3 encrypted backup must be confirmed before any script runs. If in doubt, stop and verify the backup first.

Database 1 — Pratham User Service

Tables affected (in deletion order):

  1. FieldValues
  2. UserRoleMapping
  3. UserTenantMapping
  4. Users (parent)

Database 2 — Shiksha Reports (Central)

Tables affected (in deletion order):

  1. RegistrationTracker
  2. Users (central report record)

Estimated total rows across both DBs: 25 lakh – 75 lakh rows. Scripts run in batches of 5,000 users with a 100ms pause between batches to avoid CPU spikes and WAL log bloat.

3 — Step-by-Step Execution

⚠️ Run Script 1 (Pratham) fully and verify its log before starting Script 2 (Reports). Never run both scripts simultaneously.
1
Navigate to the migration directory
cd shiksha-migration
2
Verify CSV is in place and has the correct format
head -5 users-to-delete.csv
wc -l users-to-delete.csv
Expected: first line is userId, remaining lines are UUIDs. Count should match your source report.
3
Dry-run Script 1 — Pratham User DB
node delete-users-pratham.js --dry-run
Prints pre-flight row counts for FieldValues, UserRoleMapping, UserTenantMapping, Users. No rows are deleted. Writes a log to logs/deletion-pratham-YYYY-MM-DD.json.
Verify: counts look reasonable. If any count is 0 unexpectedly, stop and investigate.
4
Live-run Script 1 — Pratham User DB
node delete-users-pratham.js --live
Executes batched deletes (5,000 users/batch) in FK-safe order. Each batch runs inside a transaction — on failure the batch rolls back and the error is logged. VACUUM ANALYZE runs automatically after all batches.
5
Verify Script 1 log
cat logs/deletion-pratham-YYYY-MM-DD.json | grep -E '"totalRowsAffected"|"error"'
Confirm totalRowsAffected matches the dry-run counts. Check that no batch has a non-null error field. If errors exist, note the batch numbers and investigate before proceeding.
6
Dry-run Script 2 — Shiksha Reports DB
node delete-users-reports.js --dry-run
Prints row counts for CourseTracker, ContentTracker, QuestionSetTracker. Writes log to logs/deletion-reports-YYYY-MM-DD.json.
7
Live-run Script 2 — Shiksha Reports DB
node delete-users-reports.js --live
Same batching and transaction safety as Script 1. VACUUM ANALYZE runs on all three tracker tables after completion.
8
Verify Script 2 log
cat logs/deletion-reports-YYYY-MM-DD.json | grep -E '"totalRowsAffected"|"error"'
Same verification as Step 5. Confirm totals and zero errors.
9
Archive logs to S3 / shared storage
Copy both JSON log files to your backup/audit storage for compliance. These logs document exactly what was deleted, in which batch, and at what time.
10
Post-deletion health check
# Check table sizes after VACUUM
SELECT relname, n_live_tup, n_dead_tup, last_vacuum, last_analyze
FROM pg_stat_user_tables
WHERE relname IN ('Users','UserTenantMapping','UserRoleMapping','FieldValues')
ORDER BY relname;
Confirm dead tuple counts are near zero after VACUUM ANALYZE.

7 — Rollback

🔴 There is no automated rollback. Deleted rows cannot be recovered from the database. Recovery requires restoring from the S3 encrypted backup.

Restore path (if required)

  1. Halt all application traffic to the affected database.
  2. Restore the S3 snapshot to a recovery instance.
  3. Verify data integrity on the recovery instance.
  4. Promote recovery instance or copy rows back selectively using the deletion log as a reference for which user IDs were affected.
  5. Resume application traffic.
⚠️ If a script run is interrupted mid-way, re-running the same script with --live is safe — already-deleted user IDs produce 0 rows affected and the script continues with remaining users.
